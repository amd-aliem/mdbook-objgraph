Rendering Engine Rules

Grid. All spacing values are multiples of the base unit (4px). Coordinates snap to the spacing scale: 4, 8, 12, 16, 20, 24, 28, 32, …

Ports. Property rows egress/ingress edges on either side (left or right). Derivation nodes egress/ingress on all four sides. When a side of a row has k connections, divide the side height into k+1 equal segments and place ports at each segment boundary. Port positions are rounded to even pixel values. Port distribution is dynamic per-row per-side.

Corridors. A corridor is the space between a node edge and its domain boundary. Every domain has a left and right corridor. An inter-domain corridor exists between every pair of adjacent domains, behaving identically to intra-domain corridors.

Channels. Each corridor contains n vertical channels. Channels have 8px padding to corridor edges and 4px padding between adjacent channels. Single-channel = 8|ch|8 = 16px. Dual-channel = 8|ch|4|ch|8 = 20px. Channel count is the minimum needed to avoid edge collisions within the corridor's y-range.

Edge routing. Edges route as orthogonal doglegs: exit horizontally from a port, travel vertically along a corridor channel, then enter horizontally at the target port. Choose the corridor side (left or right) that minimizes crossings and avoids collisions. Two edges may share a channel if their vertical extents do not overlap.

Arrowhead placement. Markers use refX=0 (arrowhead base at path endpoint). All arrowheads are 6×6. Path endpoints stop short of the target boundary by 6px. The SVG renderer draws the arrowhead forward from the path endpoint, filling the gap to the target. This prevents the thick stroke from bleeding past the arrowhead tip. All markers use markerUnits="userSpaceOnUse" for fixed pixel sizes independent of stroke width.

Selection ring. A selected node's 2px stroke ring is fully inset — the ring rect is inset by 1px so the stroke's outer edge aligns with the node rect boundary. Edge endpoints target the rect boundary and land cleanly against the ring. No overlap. Unselected nodes use a 1px light border on the rect boundary.

Derivation placement. When any input or output of a derivation crosses a domain boundary, the derivation is placed outside all domains. Derivations are centered horizontally under their input nodes when possible.

Vertical ordering. Within a column, elements follow top-down data flow: source nodes above derivations, derivations above target nodes. Inputs to a derivation are positioned above it; outputs are positioned below.

Derivation sizing. A derivation pill has no status dot — its status is communicated entirely by edge colors. Pill width accommodates text plus left/right content padding only. Text is centered within the pill.

Domain structure. Each domain has a title area (pad + cap-height + pad) followed by its content. The title-to-first-node gap equals the domain-edge-to-title pad + title cap-height + title-to-content pad. The gap after the last node to the domain bottom edge matches the inter-node gap.

State visibility principle. Problems-only. The default state has no visual indicator. Red marks appear only where something is wrong. Verified/unverified is never shown directly on a node — it is emergent from whether any of its critical properties have problem indicators.

Node → anchored state. A node is anchored when an integrity chain (e.g. signature verification) links it to a trust root. Anchored: neutral header, no dot. Unanchored: neutral header + red dot. Header color is always neutral regardless of state.

Node → selected state. A node can be selected for inspection. Selected: 2px dark stroke ring inset 1px inside the node rect boundary. Unselected: 1px light border on the boundary. Selection is orthogonal to anchored/verified state.

Property → criticality. A property is either critical (participates in the node's verified state) or informational. Critical: bold text. Informational: normal weight text. Criticality is intrinsic and does not change with state.

Property → constrained state. A critical property is constrained when at least one constraint edge targets it. Critical + constrained: bold, no dot. Critical + unconstrained: bold + red dot. Informational properties never show a dot regardless of constraint state.

Edge → validity. Every edge (anchor, constraint, derivation input/output) is either valid or invalid. Valid anchor: green solid (stroke-width 2). Valid constraint/derivation: blue solid (stroke-width 1). Invalid (any type): red dashed + red arrowhead. All arrowheads are 6×6. Edge types are distinguished by color and stroke width, not arrow size.

Derivation → state. A derivation node has no intrinsic state indicator. Its operational status is communicated entirely by the validity of its input and output edges. No dot, no color change — just the pill shape with a monospace label.
